## Context

M4.1 made the first tool real — email send via **Resend** behind a pluggable `EmailSender`, plus
idempotent execution (`GuardedExecutor` keyed on `action_id` via the audit log). **M4.2** proves the
pattern **generalizes** by adding a **second** integration — **Slack post** — reusing that exact shape:
a pluggable sender, a `build_*` factory, registry wiring, and the existing `GuardedExecutor`.

**The payoff of M4.1's design lands here for free:** because idempotency lives in
`GuardedExecutor.execute_guarded` (`src/atlas/execution.py`) for **any** `RiskTier.SEND` action,
`slack_post` inherits replay-safety with **zero new idempotency code**. M4.2 is essentially "a new
provider + a new gated tool."

**Outcome:** a real, human-gated, replay-safe `slack_post` tool (managed bot token, service identity),
provider-swappable behind a `SlackSender` boundary — dev/CI stays offline via a fake sender.

## Scope

**In scope:** `slack_post` tool (`RiskTier.SEND`); `SlackSender` ABC + `SlackApiSender`
(`slack_sdk`, managed bot token) + `FakeSlackSender`; `build_slack_sender` factory; registry wiring;
RBAC permission; config; tests; docs.

**Out of scope (deferred):** per-user Slack OAuth ("post as the user"); interactive
messages/blocks/attachments; threads, reactions, edits/deletes; Slack event/webhook ingestion;
provider-side idempotency keys. (Per-principal "send as the user" OAuth across providers is a later
phase.)

## Decisions (locked)

- **Slack via the managed `slack_sdk`** (matches the Upstash/Resend managed-service posture), behind a
  **pluggable `SlackSender`** ABC from day one.
- **Service identity** — a single workspace **bot token** (`SLACK_BOT_TOKEN`); the bot posts as itself.
  "Post as the user" (per-principal OAuth) is deferred.
- The capability is a **new, distinct** RBAC permission `tool:slack:post` (not folded into
  `tool:send`), so Slack can be granted independently. M3.5 hierarchical wildcards mean `tool:*` (and a
  future `tool:slack:*`) already satisfy it; `admin`'s `*` covers it.
- **Idempotency is inherited** from `GuardedExecutor` (audit-ledger `has_executed` by checkpointed
  `action_id`); no new idempotency code.
- **Live post requires durable Postgres audit** (`DATABASE_URL`) **and** `SLACK_BOT_TOKEN`, mirroring
  `_resolve_email_sender` — otherwise the tool is disabled (so `has_executed` can't be lost on
  restart).

## Changes

### 1. Pluggable sender — new `src/atlas/integrations/slack.py` (mirror `email.py`)

- `SlackMessage(BaseModel, frozen=True)`: `channel: str`, `text: str`. The workspace/token is owned by
  the sender, **never** by LLM tool args.
- `SlackSender(abc.ABC)`: `post(self, message: SlackMessage) -> dict[str, Any]` returning a JSON-able
  provider result (`{"ts": ..., "channel": ..., "provider": "slack"}`) so it flows straight into
  `ActionResult.output` (no `atlas_serde()` allowlist change — transient tool output, not graph state).
- `FakeSlackSender(SlackSender)`: records `sent: list[SlackMessage]`, `call_count`, optional
  `fail: bool` (raises) — the offline/test double, exactly like `FakeEmailSender`.
- `SlackApiSender(SlackSender)`: ctor `(bot_token: SecretStr)`; `post` uses **sync**
  `slack_sdk.WebClient(token=bot_token.get_secret_value()).chat_postMessage(channel=…, text=…)` —
  instance-scoped token, no global mutation (the care M4.1 took with Resend). Map
  `response["ts"]`/`response["channel"]`. On a Slack error (`SlackApiError` / `response["ok"] is
  False`) raise so `ToolRegistry.execute` records a failure. Treat the response as **data**, never
  authz. **Never log the token.** *Verify the `slack_sdk` sync `WebClient.chat_postMessage` shape via
  Context7 before coding.*
- `build_slack_sender(settings) -> SlackSender | None`: `None` unless `settings.slack_configured`;
  local `import slack_sdk` so the dep is only imported when used. Mirrors `build_email_sender`.

### 2. Tool wiring — `src/atlas/tools.py` (mirror `SendEmailTool`)

- `_SlackPostArgs(BaseModel)`: `channel: str = Field(min_length=1)`, `text: str = Field(min_length=1)`.
- `SlackPostTool(BaseTool)`: `name="slack_post"`,
  `description="Post a message to a Slack channel. Irreversible external action."`,
  `risk_tier=RiskTier.SEND`, `required_permission="tool:slack:post"`, `ArgsSchema=_SlackPostArgs`,
  ctor `(sender: SlackSender | None = None)`. `run`: type-guard args; if `sender is None` raise
  `RuntimeError("slack not configured")`; else `return sender.post(SlackMessage(channel=…, text=…))`.
- `_resolve_slack_sender(settings)`: real sender only when `settings.slack_configured` **and**
  `settings.database_url is not None` (else log the same warning style as `_resolve_email_sender` and
  return `None`).
- Register `SlackPostTool` in **both** `default_registry` (sender = `_resolve_slack_sender(settings)`)
  and `offline_registry` (sender = `FakeSlackSender()`), alongside the existing `search` + `send_email`.

### 3. RBAC — `src/atlas/governance/rbac.py` + `config/default_policies.json`

Add `"tool:slack:post"` to the `member` role's permission set (parallel to `"tool:send"`) in
**both** `ROLE_PERMISSIONS` and `config/default_policies.json` — the drift test
`test_default_policies_json_matches_role_permissions` (`tests/test_policy_store.py`) must stay green.
`admin` (`*`) already covers it; `guest` does not get it.

### 4. Config & deps — `src/atlas/config.py`, `.env.example`, `pyproject.toml`

- `slack_bot_token: SecretStr | None = Field(default=None, alias="SLACK_BOT_TOKEN")`.
- `slack_configured` property: `bool(self.slack_bot_token)`.
- `.env.example`: document `SLACK_BOT_TOKEN` (name only; it's a secret).
- `pyproject.toml`: add `slack-sdk` (import name `slack_sdk`); `uv sync`; commit `uv.lock`.

### 5. Idempotency — nothing to do (the point)

`GuardedExecutor.execute_guarded` already: for a side-effecting action (`requires_approval`), skips +
records `REPLAY_SKIPPED` when `audit.has_executed(action_id)` is true, else runs and records
`EXECUTED`/`FAILED`. `slack_post` is `RiskTier.SEND` ⇒ covered automatically. Add a **test** proving a
replayed `slack_post` posts once, but write **no** new idempotency code.

## Tests

- `tests/test_slack_integration.py` (offline): `FakeSlackSender` injected into `SlackPostTool` → `run`
  posts and returns the ts; `sender=None` → `RuntimeError("slack not configured")`;
  `build_slack_sender` returns `None` when unconfigured; `SlackApiSender` maps a stubbed
  `chat_postMessage` response (monkeypatch the `WebClient` — no network) and raises on a Slack error.
- Idempotency (reuse the M4.1 shape in `tests/test_idempotency.py` or a new test): drive
  `GuardedExecutor.execute_guarded` with an approved `slack_post` twice on one `action_id` →
  `FakeSlackSender.call_count == 1`, second pass emits `REPLAY_SKIPPED`.
- RBAC (extend the policy/graph tests): a `member` may propose `slack_post` (reaches approval); a
  `guest` is denied early (planner drops it, audit `DENIED`).
- Live Slack post gated on `-m integration` + `SLACK_BOT_TOKEN` (skipped in CI, like the Postgres /
  Upstash / Resend integration tests).
- Re-run the eval gate (`uv run python evals/run_gate.py`) — security oracles stay **1.00**.

## Security invariants (must hold)

1. `slack_post` is `RiskTier.SEND` + `required_permission="tool:slack:post"` + **human-gated**; the LLM
   picks only `channel`/`text` — never the tier, the token, or the workspace.
2. **Idempotency** via `GuardedExecutor` (no double-post on executor replay); keyed by the checkpointed
   `action_id`.
3. `SLACK_BOT_TOKEN` is a `SecretStr` sourced only from env, never logged; the Slack API response is
   treated as data, never authorization.
4. No new checkpointed types (tool output is a plain JSON-able dict in `ActionResult`); the
   `atlas_serde()` allowlist is unchanged.
5. Unconfigured ⇒ tool disabled / fail-closed (`RuntimeError`, surfaced as `ActionResult(ok=False)`);
   live post requires durable Postgres audit (matching M4.1) so idempotency survives restarts.
