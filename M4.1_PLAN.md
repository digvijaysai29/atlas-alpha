## Context

atlas takes **real, irreversible actions**, but every tool is still a **mock**
(`src/atlas/tools.py:SendEmailTool.run` returns `{"status":"sent",...}` without sending anything). M4
("real tool integrations") is large, so this first slice **M4.1** makes **one** integration real —
**email send** — and establishes the reusable pattern, plus the **idempotency** guarantee the roadmap
explicitly calls out ("avoid double-send on retry").

`send_email` is the natural first target: it is already the canonical gated tool (`RiskTier.SEND`,
`required_permission="tool:send"`, wired through approval + the hash-chained audit). M4.1 keeps all of
that and swaps the mock body for a **real send via a pluggable `EmailSender`** (Resend implemented now;
Gmail/Postmark slot in later behind the same interface), and makes execution **idempotent** so an
executor replay after a crash/resume never double-sends.

**Outcome:** a real, human-gated email send that is provider-swappable and replay-safe — with dev/CI
still fully offline (no key ⇒ dry-run mock).

## Scope

**In scope:** real email send behind an `EmailSender` ABC (Resend backend); idempotent tool execution
via a reusable execution wrapper backed by the audit log; tests; docs.

**Out of scope (deferred):** per-principal "send as the user" OAuth / credential storage (→ **M4.2**);
other integrations (Slack/Jira/Calendar); rich HTML templating; webhook ingestion; provider-side
idempotency keys (noted as future defense-in-depth).

## Decisions (locked)

- **Resend** (managed transactional-email API; API key only — same managed-service posture as the
  Upstash rate limiter from M3.6) behind a **pluggable `EmailSender`** ABC from day one.
- Send from a **verified service address** (`ATLAS_EMAIL_FROM`), e.g. `atlas@yourdomain.com`. This is
  acceptable because the send is **human-gated**; "send as the user" is M4.2.
- The capability stays provider-agnostic at the **`tool:send`** level — higher layers never know which
  provider is used.
- **Idempotency via the append-only audit log, keyed by `action_id`.** `action_id` is generated at
  **propose-time** (`actions.py:ProposedAction.action_id = _new_action_id()`) and is **checkpointed**
  in `AgentState`, so it is stable across executor restarts — exactly the idempotency-key requirement.

## Changes

### 1. Idempotency belongs in the tool-execution layer (the most important design point)

Do **not** bury the double-send check inside `make_executor_node`. Put it in a small, reusable
**execution wrapper** so the protection holds for any caller and the node stays orchestration-only.

Recommended shape — a `GuardedExecutor` (new `src/atlas/execution.py`) **or** a
`ToolRegistry.execute_guarded(action, audit)` method (reuse `src/atlas/tools.py`). It encapsulates the
side-effect rule:

```python
def execute_guarded(self, action: ProposedAction, audit: AuditLog) -> ActionResult | None:
    # Idempotency only matters for side-effecting tiers; READs are safe to re-run.
    if requires_approval(action.risk_tier) and audit.has_executed(action.action_id):
        audit.replay_skipped(action, reason="already executed")
        return None  # skipped — do not re-run the side effect
    result = self.execute(action)                  # existing registry.execute (catches errors)
    (audit.executed if result.ok else audit.failed)(result)
    return result
```

`make_executor_node` keeps **RBAC re-check + approval gating** (orchestration) and delegates
execution+idempotency+audit to this wrapper:

```python
# inside executor_node, replacing the inline execute/audit:
result = guarded.execute_guarded(action, audit)   # was: result = registry.execute(action); audit.executed(result)
if result is not None:
    results.append(result)
```

Keep `requires_approval` (`actions.py`) as the side-effect predicate (SEND/WRITE/DELETE/PAY; READ is
auto-safe). The audit `EXECUTED` event is committed durably the instant the send succeeds, so even if
the graph-state checkpoint doesn't advance before a crash, the resumed executor sees it and skips —
closing the double-send window.

### 2. Audit ledger — `src/atlas/governance/audit.py`

- Add `AuditEventType.REPLAY_SKIPPED = "replay_skipped"` and `AuditEventType.FAILED = "failed"`.
- Add recorders mirroring the existing ones: `replay_skipped(action, reason)` (→ `REPLAY_SKIPPED`) and
  `failed(result)` (→ `FAILED`, same payload shape as `executed`).
- Make `executed()` mean **success only**; route failures to `failed()` from the wrapper. (Grep tests
  /evals first — confirm nothing asserts `EXECUTED` on a *failed* action; the `agent-eval` oracles only
  assert `EXECUTED` on the success path.)
- Add `has_executed(self, action_id: str) -> bool` on the base `AuditLog`: default scans
  `self.events()` for an `EXECUTED` event with that `action_id` (success-only check ⇒ a failed send is
  **retryable**, not skipped).

### 3. Efficient Postgres check — `src/atlas/persistence/audit_store.py`

Override `has_executed` with a parameterized existence query (the table already has `event_type` and
`action_id` columns — no DDL change):

```sql
SELECT EXISTS (SELECT 1 FROM atlas_audit_log WHERE event_type = 'executed' AND action_id = %s)
```

A composite index `(action_id, event_type)` is a **later** optimization (only if audit volume grows) —
**not** part of M4.1.

### 4. Pluggable email sender — `src/atlas/integrations/email.py` (new)

- `EmailMessage(BaseModel, frozen=True)`: `to: str`, `subject: str`, and **both** `html: str | None`
  and `text: str | None` bodies (support both even if one is primary). The `from` address is owned by
  the sender, **never** by the LLM/caller.
- `EmailSender(abc.ABC)`: `send(self, message: EmailMessage) -> dict[str, Any]` returning a JSON-able
  result (`{"id": <provider id>, "provider": "resend", "to": ...}`) so it flows straight into
  `ActionResult.output` (no `atlas_serde()` allowlist change — tool output is transient data, not graph
  state).
- `ResendEmailSender(EmailSender)`: ctor `(api_key, from_address)`; `send` uses the **sync** Resend
  SDK — `resend.api_key = ...; resend.Emails.send({"from","to","subject","html"/"text"})` → maps the
  returned `id`. **Verify the exact SDK shape via Context7 before coding.** Never log the key; treat
  the provider response as **data** (adversarial), never authorization.
- `build_email_sender(settings) -> EmailSender | None`: `None` unless `settings.email_configured`
  (key + from set); local `import resend` so the SDK is only required when used. Mirrors
  `build_rate_limiter` / `build_authenticator`.
- `src/atlas/integrations/__init__.py` re-exports `EmailSender`, `EmailMessage`, `build_email_sender`.

### 5. Wire the real tool — `src/atlas/tools.py` + `src/atlas/orchestration/graph.py`

- `SendEmailTool.__init__(self, sender: EmailSender | None = None)`. `run(args)`: when `sender` is set,
  build an `EmailMessage` and `return sender.send(...)`; when `None`, return the existing **mock /
  dry-run** dict (keeps dev/CI offline). `risk_tier` + `required_permission` unchanged.
- `default_registry(settings: Settings | None = None)`: build the sender via `build_email_sender` and
  pass it to `SendEmailTool(sender=...)`.
- `graph.py:build_graph` already does `registry = registry or default_registry()` → change to
  `default_registry(settings)` (settings is in scope). Thread the `GuardedExecutor`/`execute_guarded`
  into `make_executor_node` alongside the existing `registry, audit, policy`.

### 6. Config & deps — `src/atlas/config.py`, `.env.example`, `pyproject.toml`

- `resend_api_key: SecretStr | None` (`RESEND_API_KEY`), `email_from: str | None`
  (`ATLAS_EMAIL_FROM`); `email_configured` property = `bool(resend_api_key and email_from)`.
- `.env.example`: document the two names only (no values; the key is secret).
- `pyproject.toml`: add `resend`; `uv sync`; commit the lockfile.

### 7. Tests

- `tests/test_email_integration.py` (offline): a fake `EmailSender` injected into `SendEmailTool`
  records the `EmailMessage` and returns an id → `run` returns it; `sender=None` → dry-run mock dict;
  `build_email_sender` returns `None` when unconfigured. **Live** Resend send gated on `-m integration`
  + `RESEND_API_KEY`/`ATLAS_EMAIL_FROM` (skipped in CI, like the Postgres/Upstash patterns).
- `tests/test_idempotency.py` (offline): drive `execute_guarded` (or the executor node) with an
  `InMemoryAuditLog`, an approved SEND action, and a **counting** fake sender via the registry; run it
  **twice** with the same checkpointed `action_id` → sender called **once**, the second pass emits
  `REPLAY_SKIPPED` and no second `EXECUTED`; a **failed** first send records `FAILED` (not `EXECUTED`)
  and is **retried** on the next pass (`has_executed` stays False). Unit-test `has_executed` both ways.
- `tests/test_audit_postgres.py` (`-m integration`): `has_executed` round-trips via the SQL `EXISTS`.
- Re-run the **eval gate** — `approval/approve` still sees `EXECUTED`; no oracle asserts `EXECUTED` on
  a failed action.

## Security invariants (must hold)

1. Email send stays `RiskTier.SEND` + `required_permission="tool:send"` + **human-gated**; the LLM
   only picks tool+args — never the tier, never the from-address.
2. **Idempotency** prevents double side-effects on executor replay (audit ledger keyed by `action_id`),
   and it lives in the **execution layer**, not the node, so any future call path is protected.
3. `RESEND_API_KEY` is a `SecretStr` sourced only from env, never logged; provider responses are data,
   never authorization.
4. No new checkpointed types (tool output is a plain JSON-able dict in `ActionResult`); the
   `atlas_serde()` allowlist is unchanged.
5. A send failure → `FAILED` + `ActionResult(ok=False)`, **not** `EXECUTED`, so it is retryable and
   never silently swallowed. Unconfigured ⇒ dry-run mock (dev/CI stays offline).
