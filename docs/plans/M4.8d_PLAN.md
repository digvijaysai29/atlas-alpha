# M4.8d — New Connector via Schema + `/approve/stream` + Token-Level Streaming

**Status:** implemented.
**Prereqs:** M4.8a adapter engine, M4.3 per-user OAuth (`CredentialResolver`), M4.7 SSE streaming
(`/chat/stream`, `interface/sse.py`), M4.5 OpenRouter provider pattern (`knowledge/extraction.py`).
**Companion docs:** [`CUSTOM_CONNECTORS.md`](../guides/CUSTOM_CONNECTORS.md),
[`RUNBOOK.md`](../guides/RUNBOOK.md), [`M4.8c_PLAN.md`](M4.8c_PLAN.md).

## Context

Three things M4.8a/M4.7 explicitly deferred, closed together:

1. **A genuinely new connector, purely via schema** — M4.8a proved the engine by *re-expressing* an
   existing tool (`slack_post_as_user`); nothing new was added. M4.8d adds one that didn't exist
   before, using only a JSON file.
2. **`/approve/stream`** — M4.7 shipped `/chat/stream` but deferred the resume sibling because it
   "would need `verify_thread_owner` before the body streams." That's now solved.
3. **Token-level streaming** — M4.7's responder was a deterministic string formatter with no LLM
   call, so there was nothing to stream. M4.8d gives it one.

## Decisions (locked)

| Decision | Choice |
|---|---|
| New connector | `slack_delete_message` (Slack `chat.delete`), `DELETE` risk tier, reuses the `chat:write` scope already granted for `slack_post_as_user` — **zero** new OAuth/provider code. Flat `{channel, ts}` payload fits the engine's existing top-level-field mapping; `slack.com` was already egress-allowlisted. |
| Not resource-scoped | Consistent with M4.8c's decision to leave schema-driven tools unscoped this round. |
| `/approve/stream` ordering | Fetch snapshot → 404 if missing → `verify_thread_owner` (403) → awaiting-approval check (409) → stream. Identical order to sync `/approve`, preserved for the same anti-enumeration reason. |
| Streaming transport | Shared `_stream_lifecycle` generator (extracted from M4.7's `_chat_event_stream`) used by both `/chat/stream` and `/approve/stream` — `run_graph_stream(atlas.graph, resumable, config, send_stream, stream_modes=("updates","messages"))`; `resumable` is either a fresh `initial_state(...)` or a `Command(resume=decisions)` — `graph.stream` accepts both, unchanged. |
| Token streaming provider | **OpenRouter**, not direct `ChatAnthropic` — mirrors `knowledge/extraction.py`'s established pattern (primary model + `.with_fallbacks` chain) via a new, standalone `ResponderLLM` class rather than the planner's older direct-Anthropic `llm.py:build_model()`. Deliberate, discussed scope boundary: `llm.py`/`llm_plan` (the planner) is **not touched at all** this milestone — left for a separate, dedicated change given how heavily-tested that path is. |
| Responder default | Deterministic `_summarize()` unless `ATLAS_RESPONDER_LLM_ENABLED=true` **and** `OPENROUTER_API_KEY` is set (reuses the same key as M4.5 extraction) — off by default, so CI/the eval gate stay hermetic. On any LLM failure, falls back to the already-computed deterministic summary rather than failing the turn. |
| `token` event scope | Filtered to `metadata["langgraph_node"] == "responder"` only — planner tool-call-argument deltas are never forwarded as SSE tokens (would leak partial/malformed JSON as if it were the answer). |

## How token streaming actually works (verified against the installed packages, not assumed)

- `graph.stream(state_or_command, config, stream_mode=["updates", "messages"])` yields `(mode, data)`
  2-tuples; `"messages"` data is `(message_chunk, metadata)`, `metadata["langgraph_node"]` names the
  emitting node. Confirmed by reading `langgraph.pregel.main._output` directly (dispatches on
  `isinstance(stream_mode, list)` — **not** element count, so a single-mode list would also change
  shape; `run_graph_stream` therefore passes a bare string for one mode and a list only for several).
- `.invoke()` on a chat model still produces token-by-token callback events — the mechanism
  `stream_mode="messages"` taps into — as long as `_should_stream` returns true, most reliably via
  `streaming=True` at construction (confirmed in `langchain_core.language_models.chat_models` and the
  installed `ChatOpenRouter` — it declares a `streaming: bool` field). The responder node itself calls
  `.invoke()` exactly like `llm_plan` already does; no manual chunk-collection loop needed in node code.

## Changes

1. `src/atlas/tool_schemas/slack_delete_message.json` (new).
2. `src/atlas/orchestration/responder_llm.py` (new): `ResponderNarrator` (ABC), `ResponderLLM`
   (OpenRouter, primary + fallback, `streaming=True`, lazy + memoized), `FakeResponderLLM` (test
   double), `make_responder_llm(settings)`.
3. `src/atlas/config.py`: `ATLAS_RESPONDER_LLM_ENABLED` / `ATLAS_RESPONDER_MODEL` /
   `ATLAS_RESPONDER_FALLBACK_MODELS` + `responder_llm_active` / `responder_fallback_model_list` +
   all-or-nothing validator (mirrors `extraction`).
4. `src/atlas/orchestration/nodes.py`: `make_responder_node(settings, *, responder_llm=None)` —
   deterministic summary computed first (always), optionally replaced by the narrator, try/except
   fallback on any failure. `src/atlas/orchestration/graph.py` passes `settings` through.
5. `src/atlas/interface/sse.py`: `TOKEN` event constant; `run_graph_stream(..., stream_modes=(...))`.
6. `src/atlas/interface/routes.py`: `_chunk_text` (defensive content extraction); shared
   `_stream_lifecycle` (replaces `_chat_event_stream`); new `POST /approve/stream`.
7. `config/default_policies.json` / `governance/rbac.py`: `tool:slack:delete_message` granted to
   `member` (bundled into the M4.8c commit alongside the other policy migration).

## Out of scope (future)

Migrating the planner's direct `ChatAnthropic` (`llm.py`, `llm_plan`) to the same OpenRouter +
fallback pattern — a deliberate, separate follow-on given it's the most heavily-tested LLM call site
in the system; SSE for `/threads/{id}` tailing; nested/JSONPath payload mapping in the adapter engine
(would be needed for APIs like Google Calendar's nested `start`/`end` objects).
