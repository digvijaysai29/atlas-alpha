# M4.8c ‚Äî Resource/Argument-Aware `ToolPermission`

**Status:** implemented.
**Prereqs:** M2.2a RBAC (`governance/rbac.py`), M3.4 pluggable `PolicyStore`, M3.5 hierarchical wildcard
matching.
**Companion docs:** [`AUTH.md`](../guides/AUTH.md), [`CLAUDE.md`](../constitution/CLAUDE.md).

## Context

`required_permission` has always been a flat string per tool (e.g. `"tool:slack:post"`) ‚Äî a role
granted that string could post to *any* Slack channel, or (via `send_email`/`gmail_send`) email
*any* recipient. `AUTH.md`'s deferred-work table has carried "resource/argument-aware RBAC" since
M3.3 as a placeholder. M4.8c closes it: the permission a caller must hold can now be scoped by an
argument of the action itself ‚Äî channel, recipient domain ‚Äî without inventing a new authorization
model.

## Decisions (locked)

| Decision | Choice |
|---|---|
| Matching engine | **Unchanged.** `governance/rbac.py:permission_satisfied` already handles arbitrary colon-segmented strings + trailing `:*` wildcards. A resource-scoped permission is just a longer string; no new matching logic. |
| Where the scope comes from | `BaseTool.resource_permission(args) -> str \| None`, an overridable hook (default `None`) analogous to the existing `audit_metadata()` hook ‚Äî never the LLM. |
| When it's computed | **Once**, in `ToolRegistry.propose()`, from the tool's base `required_permission` + `resource_permission(validated_args)`, stamped onto the immutable `ProposedAction.required_permission`. The planner and executor both read that field ‚Äî never re-derive it ‚Äî so the two RBAC checks can never disagree. |
| Scoped this milestone | `slack_post` ‚Üí `channel:<name-or-id>`; `send_email` / `gmail_send` ‚Üí `domain:<lowercased recipient domain>`. |
| Deliberately NOT scoped | `slack_post_as_user` (has a schema-driven twin in the adapter engine; M4.8a's hand-written/schema equivalence test asserts identical behavior ‚Äî scoping only the hand-written side would break that) and all adapter-engine schema tools (`ToolSchema.required_permission` stays a flat string). Revisit once a schema needs it. |
| Default-policy migration | `ROLE_PERMISSIONS["member"]` / `config/default_policies.json`: `tool:send` ‚Üí `tool:send:*`, `tool:slack:post` ‚Üí `tool:slack:post:*`, `tool:gmail:send` ‚Üí `tool:gmail:send:*` (preserves today's "any recipient/any channel" default). Also grants `tool:slack:delete_message` (M4.8d). |

## Changes

1. `src/atlas/actions.py`: `ProposedAction.required_permission: str | None` (new field).
2. `src/atlas/tools.py`: `BaseTool.resource_permission()` hook; overrides on `SlackPostTool`
   (`channel:...`, strips only a leading `#`, no case-folding), `SendEmailTool` / `GmailSendTool`
   (`domain:...`, lowercased); `ToolRegistry.propose()` stamps the combined permission via a new
   `_combine_permission` helper.
3. `src/atlas/orchestration/nodes.py`: `make_planner_node` / `make_executor_node` read
   `action.required_permission` instead of re-deriving from `registry.get(action.tool)`.
4. `src/atlas/governance/rbac.py` + `config/default_policies.json`: wildcard migration (above).

## Operator impact

A Postgres `atlas_role_permissions` table seeded **before** this change still holds the old bare
grants (`tool:slack:post`, etc.). Those no longer satisfy the now resource-scoped required
permissions (a bare grant has no trailing `:*`, so `permission_satisfied` denies). The existing
`missing_default_grants()` startup warning (`make_policy_store`) already surfaces this automatically;
`scripts/manage_policy.py seed` is additive (`INSERT ... ON CONFLICT DO NOTHING`), so re-running it
after upgrading is safe and sufficient ‚Äî no new migration tooling needed.

**New capability this unlocks:** an operator can grant a *narrower* permission instead of the
wildcard ‚Äî e.g. `manage_policy.py grant member tool:gmail:send:domain:company.com` (after revoking the
wildcard) restricts a role to emailing only that domain. See `AUTH.md` for the full example.

## Tests

- `tests/test_tools.py`: `resource_permission` derivation (channel, domain, case-normalization) and
  `ProposedAction.required_permission` stamping; confirms `slack_post_as_user` and `search` stay
  unscoped; confirms extra args can't smuggle a different scope.
- `tests/test_rbac.py`: wildcard-covers-resource-scoped-string; wildcard-does-NOT-cover-the-bare-
  unscoped-string; a narrowed single-domain grant allows the matching domain and denies another;
  end-to-end ‚Äî a role granted only `tool:slack:post:channel:general` is denied (before approval) for
  any other channel.
- `tests/test_policy_store.py`: updated two pre-existing assertions that hardcoded the old bare
  `tool:send` string against the now-wildcarded default policy.
- `tests/test_adapter_engine.py`: unchanged, verified green (regression guard for the "schema tools
  stay unscoped" decision).
