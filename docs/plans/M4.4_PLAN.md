## Context

atlas's knowledge layer was architecturally complete but **inert**: the `KnowledgeGraph` interface,
RBAC read-filtering (`can_read`), and the durable `PostgresKnowledgeGraph` full-text backend all
existed, but the **only** way entities entered the graph was hand-written demo scripts. So the
"compounding Personal/Organizational Knowledge Graph" the constitution names as the product moat
(§1) accumulated nothing.

**M4.4** builds the smallest shippable slice that makes the PKG/OKG actually populate: a
**deterministic, hermetic** ingestion pipeline (document → chunks → idempotent, scope/ACL-tagged
`Entity` upserts) behind a trusted `POST /kg/ingest` endpoint, with **per-user PKG read isolation**
and a **fail-closed OKG write gate**. It is intentionally deterministic (no LLM, no embeddings, no
external connectors) so the whole write path is covered by the blocking deterministic eval gate.

**Outcome:** documents ingested via `/kg/ingest` immediately become RBAC-scoped grounding the
planner retrieves through its existing `query` — the agent's memory starts compounding.

## Scope

**In scope:** deterministic `IngestionService` (chunk → entity → idempotent upsert); `POST /kg/ingest`;
per-user PKG isolation via **identity ACLs**; fail-closed OKG write gate (`kg:write:org`);
content-agnostic audit event; deterministic eval oracles; tests.

**Out of scope (deferred):** LLM entity/relation extraction (M4.5); pgvector embeddings + semantic
retrieval (M4.6); OAuth-connector pull (Gmail/Slack/Calendar sync) — the same `IngestionService` will
back it; bulk/file upload; relation discovery; `delete_by_source` / doc-supersession.

## Decisions (locked)

- **Deterministic & hermetic chunking** via a small in-module fixed-window splitter
  (`chunk_text`, `DEFAULT_CHUNK_SIZE=2000`, `DEFAULT_CHUNK_OVERLAP=200`). `langchain_text_splitters`
  is not installed and a fixed-window chunker is a pure function (not infra) — so no new dependency,
  and the eval gate covers it deterministically. Boundary-aware/semantic chunking is a later upgrade.
- **Deterministic entity ids = idempotency.** `id = f"{owner_segment}:{source_id}:{i:04d}"` where
  `owner_segment` is `personal:{user_id}` or `org`, and `source_id` is caller-provided or
  `sha256(title)[:16]`. Re-ingesting the same `(scope, owner, source_id)` upserts the same chunk ids
  via the existing `_UPSERT_ENTITY` `ON CONFLICT` — **no duplicate growth**, and two users ingesting
  the same title never collide. Known v1 limitation: an *edited* doc with fewer chunks can leave
  stale trailing chunks (no `delete` on the interface) — deferred to a `delete_by_source` follow-up.
- **PKG isolation via identity ACLs.** `personal` ingest stamps `acl=(kg:read:user:{user_id},)`.
  `can_read` honors a `kg:read:user:<uid>` entry **only** by exact `user_id` match — never via a role
  wildcard (`*` / `kg:read:*`) — so a PKG stays private even from `admin`. Both backends agree:
  `PostgresKnowledgeGraph.query` injects the principal's own identity grant into the bound `exact`
  array (no SQL string change), and the Python `can_read` re-filter remains the authority.
- **OKG write is fail-closed.** `org` ingest requires `policy.can(principal, "kg:write:org")`. That
  permission is satisfied only by `admin`'s `*` today (no role holds a `kg:*`/`kg:write:*` grant), so
  it is admin-only by default without adding a redundant `ROLE_PERMISSIONS` entry; a future role can
  be granted it through the `PolicyStore`. Anonymous principal, unknown scope, or `org_acl` on a
  `personal` request → `IngestionDenied` (nothing written).
- **No new checkpointed state / no serde change.** Ingestion is a synchronous API action that reuses
  the existing `Entity` upsert path; the `atlas_serde()` allowlist is untouched (mirrors M4.2).
- **Single source of truth.** `make_ingestion_service` builds the service over the *same* KG, policy,
  and audit `build_graph` already constructs; it's exposed on the shared `Atlas` (`app.state`).

## Changes

1. **`src/atlas/knowledge/ingestion.py` (new):** `IngestDocument`, `IngestionResult`,
   `IngestionDenied`, `chunk_text`, and `IngestionService.ingest` (validate principal → resolve
   scope/ACL → chunk → idempotent upsert → content-free audit `ingested`).
2. **`src/atlas/knowledge/interfaces.py`:** `IDENTITY_ACL_PREFIX` + `identity_acl()`; `can_read`
   extended to honor identity ACLs (exact owner match, never via wildcards) — additive.
3. **`src/atlas/persistence/knowledge_store.py`:** `query` injects `identity_acl(principal.user_id)`
   into the `exact` overlap param (static SQL unchanged).
4. **`src/atlas/governance/audit.py`:** new `AuditEventType.INGESTED` + `AuditLog.ingested()` recorder
   (counts/scope/actor only — never document content).
5. **`src/atlas/orchestration/graph.py`:** `make_ingestion_service` factory + `Atlas.ingestion` field.
6. **`src/atlas/interface/{kg_routes.py (new),schemas.py,app.py}`:** `POST /kg/ingest`,
   `IngestRequest`/`IngestResponse`, router registration; `IngestionDenied → 403`, validation → 422.

## Security invariants (must hold)

1. **OKG write is fail-closed** — `scope=org` requires `kg:write:org`; member/guest/anonymous denied,
   nothing persisted.
2. **PKG is per-user** — a `personal` entity is readable only by the owning `user_id`, enforced in SQL
   **and** re-checked by `can_read` (defense-in-depth); not even `admin` reads another's PKG.
3. **No model-driven authz** — scope/ACL derive from the authenticated principal + server rules, never
   from document content (the caller only *requests* a scope, which the server validates).
4. **Idempotent, append-safe** — deterministic ids mean replays upsert in place; the append-only audit
   records each ingest; no checkpointed-type / serde changes.
5. **No content leakage** — audit + logs carry counts/scope/source_id/actor only; parameterized SQL.
