## Context

The knowledge layer compounds (M4.4 made the write path real) but retrieval was **lexical only**:
`PostgresKnowledgeGraph` ranked entities with `tsvector` full-text + an `ILIKE` substring fallback. The
agent could not find a relevant note that shared no keywords with the query. M4.6 adds **semantic
retrieval** on the existing Neon Postgres via **pgvector**, behind the same `KnowledgeGraph` interface,
**without weakening RBAC scoping** — the security spine of the PKG/OKG.

**Outcome:** entities are embedded on upsert; `query` runs **hybrid** retrieval (full-text + vector
cosine, fused with Reciprocal Rank Fusion) for both the Personal and Organizational graphs, with the
identical RBAC predicate on both branches.

## Decisions (locked)

- **Hybrid, not vector-only.** Keep the proven FTS path; add a vector branch; fuse the two rankings with
  **RRF** (`k=60`). When there is no embedder or the query is empty (match-all), behavior is exactly the
  M3.1 full-text path — full back-compat.
- **Pluggable embedder, hermetic by default.** `VoyageEmbedder` (Voyage AI, Anthropic's recommended
  partner; `voyage-3`, 1024-dim) is used **only** when `VOYAGE_API_KEY` is set. Otherwise a
  `DeterministicEmbedder` (pure, hash-seeded, L2-normalized — no network) is used, so CI and the
  deterministic eval gate stay hermetic and free. A single `ATLAS_EMBEDDING_DIM` drives both the column
  width and every embedder, so a wrong-width vector can never be written.
- **Embedding is a store side effect.** `PostgresKnowledgeGraph.upsert_entity` computes the embedding
  from `name + "\n" + content`; `IngestionService` stays deterministic (no embeddings in the eval gate).
- **In-memory stub stays keyword-only.** Semantic retrieval is a Postgres-backend capability, so the
  offline stub (and thus the eval gate) is untouched — the gate remains **1.00**.
- **Neon.** Reuses the existing project `atlas-alphs` (`small-shadow-32816321`, PG18). `pgvector` 0.8.1
  is enabled; the `embedding vector(1024)` column + HNSW (`vector_cosine_ops`) index are created
  idempotently by `setup()`.

## Tests

- **`tests/test_embeddings.py` (offline):** deterministic embedder (reproducible, dim, L2-norm, distinct
  text, batch); `make_embedder` selection (Voyage vs deterministic); blank-key rejection; `_vector_literal`;
  RRF fusion; **`test_rbac_predicate_is_shared_by_both_query_branches`** (the predicate is byte-identical
  in `_QUERY` and `_VECTOR_QUERY` — drift guard).
- **`tests/test_knowledge_postgres.py` (integration, Neon):** embedding populated on upsert; hybrid query
  hides the org doc from a guest / shows it to a member (**IDOR guard on the vector path**); vector branch
  ranks an exact embedded-text query first; FTS fallback when the store has no embedder.

## Security invariants (must hold)

1. **No IDOR via embeddings** — the FTS and vector branches share the *identical* RBAC predicate
   (enforced by a test); `can_read` is the final authority. Semantic search cannot surface a row keyword
   search could not.
2. **PKG/OKG scoping holds on the vector path** — a guest never retrieves an org doc semantically; not
   even `admin` reads another user's PKG.
3. **No model-driven authz** — scope/ACL derive from the principal; the query vector only affects ranking.
4. **No content/secret leakage** — embedding source text is never logged; the Voyage key comes only from
   env and is never logged; all SQL is static + parameterized (the vector is a bound `::vector` literal).
5. **Hermetic by default** — no API key ⇒ deterministic embedder; the eval gate stays **1.00**.
