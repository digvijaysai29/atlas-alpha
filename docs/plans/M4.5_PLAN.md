## Context

M4.4 made the knowledge-graph *write path* real but deterministic: a document is chunked into fixed
windows and stored as `type="doc"` chunk entities — a searchable bag of text, never an actual *graph*.
M4.5 enriches that path: an LLM reads each document and proposes **typed concept entities**
(person / project / org / concept) and **directed relations** between them, written alongside the
chunks so the graph becomes navigable and the planner can reason over structure, not just keyword/
vector similarity.

**Outcome:** with extraction enabled, an ingested document yields its deterministic chunks **plus**
typed concept nodes, inter-entity edges (e.g. `person --works_on--> project`), and `mentions` edges
anchoring each concept to the source. With extraction disabled (the default) the write path is
byte-for-byte M4.4.

## Decisions (locked)

- **Opt-in, hermetic by default.** Extraction runs only when `ATLAS_KG_EXTRACTION_ENABLED=true` **and**
  `OPENROUTER_API_KEY` is set; otherwise a `DeterministicExtractor` (pure no-op returning an empty
  result) is used, so CI and the blocking deterministic eval gate stay hermetic at **1.00** and ingest
  incurs no LLM cost. A `@model_validator` rejects the flag-on-without-key combination (fail-fast).
- **OpenRouter, not a single provider.** `LLMExtractor` calls OpenRouter's OpenAI-compatible endpoint
  via LangChain `ChatOpenAI`, composing a **primary model + fallback chain** with `.with_fallbacks([...])`
  and `.with_structured_output(ExtractionResult)`. Models are configured by id
  (`ATLAS_EXTRACTION_MODEL` + comma-separated `ATLAS_EXTRACTION_FALLBACK_MODELS`); the SDK is lazy-imported.
- **The model never sets authorization.** Extracted nodes inherit the **server-resolved** `scope`/`acl`
  from `IngestionService._resolve_scope_acl(principal, document)` verbatim — never from the document or
  the model output. The model decides only *which* nodes/edges exist (mirrors M4.6: the query vector
  affects ranking only).
- **Untrusted output is bounded.** Every node/edge is validated against a strict Pydantic schema (closed
  `EntityKind` Literal); counts are capped (`ATLAS_EXTRACTION_MAX_ENTITIES`/`_RELATIONS`); duplicates are
  deduped by `(kind, normalized name)`; a relation is dropped unless **both** endpoints resolve to an
  extracted entity (no dangling/hallucinated edges).
- **Enrichment never breaks the core write.** Extraction runs *after* the chunk writes; any failure is
  logged by exception **type only** (no content) and degrades to chunks-only.
- **Deterministic, idempotent ids.** Extracted entity id =
  `{owner_segment}:{source_id}:entity:{kind}:{sha256(kind+name)[:12]}` — namespaced for PKG isolation,
  collision-resistant, so re-ingesting upserts rather than duplicates.

## Changes

1. **`src/atlas/knowledge/extraction.py` (new):** `ExtractedEntity`/`ExtractedRelation`/`ExtractionResult`
   schemas, `EntityExtractor` ABC, `DeterministicExtractor` (no-op), `FakeExtractor` (tests),
   `LLMExtractor` (OpenRouter + fallbacks + structured output, lazy SDK), `make_extractor(settings)`.
2. **`src/atlas/config.py`:** `OPENROUTER_API_KEY`, `ATLAS_KG_EXTRACTION_ENABLED`, `ATLAS_EXTRACTION_MODEL`,
   `ATLAS_EXTRACTION_FALLBACK_MODELS`, `ATLAS_EXTRACTION_MAX_ENTITIES`/`_RELATIONS`; `openrouter_configured`,
   `extraction_enabled`, `extraction_fallback_model_list` properties; `validate_extraction_config`.
3. **`src/atlas/knowledge/ingestion.py`:** optional `extractor` + cap params on `IngestionService`;
   `_enrich_with_extraction` / `_write_relations` (server-resolved scope/acl, dedup, dangling-edge drop,
   caps, `mentions` anchoring, degrade-on-failure); `IngestionResult` gains `extracted_entity_count` /
   `relation_count`.
4. **`src/atlas/orchestration/graph.py`:** `make_ingestion_service` builds `make_extractor(settings)` and
   injects it + caps.
5. **`src/atlas/governance/audit.py`:** `ingested(...)` records extracted-entity/relation counts (counts
   only, added only when non-zero — a no-extraction ingest is byte-identical to M4.4).
6. **`pyproject.toml`:** `langchain-openai`.
7. **Docs:** this plan; `ARCHITECTURE.md` + `CLAUDE.md` roadmaps; `.env.example` (extraction vars).

## Tests

- **`tests/test_extraction.py` (offline):** schema rejects unknown kind / blank name; `make_extractor`
  selection (flag+key ⇒ `LLMExtractor`, else `DeterministicExtractor`); flag-without-key rejected;
  cap validation; fallback-list parsing; `LLMExtractor` blank-key/blank-text guards (no network);
  end-to-end through `IngestionService` with `FakeExtractor`: typed entities + relations written,
  **scope/acl inherited from principal not content** (prompt-injection guard), org-scope ACL when
  authorized, idempotent re-ingest, dedup, dangling-relation drop, entity/relation caps, degrade-on-
  failure, content-free audit counts.
- **`tests/test_knowledge_postgres.py` (integration, pgvector Postgres):** extracted entities + relations
  persist and survive a fresh store instance; **IDOR guard on extracted nodes** (personal concepts hidden
  from another user and from admin's wildcard; org concepts visible to org readers, hidden from guest);
  re-ingest does not duplicate extracted nodes.

## Security invariants (must hold)

1. **No model-driven authz** — extracted node `scope`/`acl` come only from `_resolve_scope_acl(principal)`;
   the model/document never set them. A personal ingest of a document claiming `org` still stamps the
   caller's identity ACL (tested).
2. **PKG/OKG scoping holds on extracted nodes** — a guest never reads org-scoped concepts; not even
   `admin` reads another user's personal concepts (identity ACL).
3. **Untrusted output is validated + bounded** — strict schema, count caps, dedup, dangling-edge drop.
4. **No content/secret leakage** — document text is never logged or audited (counts/scope only); the
   `OPENROUTER_API_KEY` comes only from env and is never logged.
5. **Hermetic by default** — no flag/key ⇒ deterministic no-op extractor; the eval gate stays **1.00**.
