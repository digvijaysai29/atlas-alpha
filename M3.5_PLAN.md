## Context

Authorization today matches permissions **exactly**, plus a single global `"*"` admin wildcard
(`src/atlas/governance/policy.py:PolicyStore.can` and `src/atlas/governance/rbac.py:can`). Permission
strings are already colon-segmented (`tool:send`, `kg:read:org`, `kg:read:personal`), but a role must
be granted each leaf explicitly. M3.5 adds **hierarchical wildcard grants**: a granted `kg:read:*`
satisfies a required `kg:read:org`; `tool:*` satisfies any `tool:...`. Permissions stay plain strings
— **no** new data model, ABAC, or resource/argument-aware logic (all deferred).

**Outcome:** policies become far more usable (grant `kg:read:*` once instead of every leaf) while
preserving default-deny / fail-closed semantics and the "the LLM can never self-grant" invariant.

## Scope

**In scope:** wildcard *matching* on the granted side, applied consistently across
`InMemoryPolicyStore`, `PostgresPolicyStore`, the Postgres KG SQL read filter, and `can_read`; tests;
docs.

**Out of scope (deferred):** resource/argument-aware permissions ("only send to internal domains"),
structured `ToolPermission` objects, ABAC, cross-role permission inheritance, and per-principal rate
limiting (→ **M3.6**).

## Wildcard semantics (implement exactly this)

A **granted** permission `g` satisfies a **required** permission `r` iff:

1. `g == "*"` (global admin wildcard — unchanged), OR
2. `g == r` (exact match — unchanged), OR
3. `g` ends with `":*"` **and** `r.startswith(g[:-1])` — where `g[:-1]` is the prefix **including the
   trailing colon**. Examples: `kg:read:*` (prefix `kg:read:`) matches `kg:read:org` and
   `kg:read:personal`; `tool:*` (prefix `tool:`) matches `tool:send`; `kg:*` matches `kg:read:org`.

Rules / non-goals to preserve:

- Wildcards are interpreted **only on the granted side**. The *required* permission (a tool's
  `required_permission`) is always concrete — never interpret `*` in it.
- A granted `kg:read` (no trailing `:*`) does **not** implicitly cover `kg:read:org` — only an
  explicit `:*` suffix expands. (No silent hierarchy.)
- Fail-closed unchanged: `None` principal / empty grant set / unknown role → deny.

## Changes

### 1. `src/atlas/governance/rbac.py` — the matching helper (single source of truth)

Add a pure function (mirrors the existing `expand_roles`):

```python
def permission_satisfied(granted: frozenset[str], required: str) -> bool:
    """True iff any granted permission satisfies `required` (exact, global "*", or "a:b:*" prefix)."""
    if WILDCARD in granted or required in granted:
        return True
    return any(g.endswith(":*") and required.startswith(g[:-1]) for g in granted)
```

Refactor the free `can` to use it:

```python
def can(principal, permission):
    if permission is None:
        return True
    return permission_satisfied(get_effective_permissions(principal), permission)
```

Export `permission_satisfied` from `src/atlas/governance/__init__.py` (keep the public surface tidy).

### 2. `src/atlas/governance/policy.py` — base `PolicyStore.can`

Replace the body of `PolicyStore.can` to delegate to the helper (every backend inherits it):

```python
def can(self, principal, permission):
    if permission is None:
        return True
    return permission_satisfied(self.effective_permissions(principal), permission)
```

Import `permission_satisfied` from `atlas.governance.rbac` (one-way; no cycle). `InMemoryPolicyStore`
needs no change — it already returns the granted set, which may now contain `:*` patterns.

### 3. `src/atlas/persistence/knowledge_store.py` — wildcard-aware SQL filter (THE tricky part)

The KG read filter currently uses array overlap `acl && %(perms)s::text[]`, which is **exact match
only**. With wildcard grants this would **under-include** — silently hide entities a `kg:read:*` grant
should reveal. That is a correctness bug the Python `can_read` re-filter cannot fix (it can't add back
rows the SQL never fetched). So the SQL itself must become wildcard-aware.

In `query`, split the granted set:

```python
permissions = self._policy.effective_permissions(principal)
is_admin = _ADMIN_WILDCARD in permissions
exact = [p for p in permissions if p != _ADMIN_WILDCARD and not p.endswith(":*")]
# "kg:read:*" -> LIKE pattern "kg:read:%", with %/_ in the prefix escaped via _like_escape
wildcard_like = [f"{_like_escape(p[:-1])}%" for p in permissions if p.endswith(":*")]
```

Extend the acl clause in `_QUERY` (still fully parameterized, static SQL):

```sql
(
    %(is_admin)s = TRUE
    OR acl IS NULL
    OR cardinality(acl) = 0
    OR acl && %(exact)s::text[]
    OR EXISTS (SELECT 1 FROM unnest(acl) AS a WHERE a LIKE ANY(%(wildcard_like)s::text[]))
)
```

Pass `exact` and `wildcard_like` in `params` (drop the old `perms`). `LIKE ANY('{}')` is harmless
(false) when there are no wildcard grants. The Python defense-in-depth re-filter stays
`can_read(principal, entity, self._policy)` — now wildcard-aware via the shared helper, so it agrees
with the SQL (backend parity). **A test must prove `kg:read:*` reveals an org entity over Postgres.**

### 4. CLI / seeding — `scripts/manage_policy.py`, `config/default_policies.json`

## Security invariants (must hold)

1. Default-deny / fail-closed unchanged; wildcards only **grant**, never widen the **required** side.
2. **Backend parity:** `InMemoryPolicyStore`, `PostgresPolicyStore`, the Postgres KG SQL filter, and
   the Python `can_read` re-filter must all agree on wildcard matching (shared `permission_satisfied`).
3. **No KG IDOR regression:** the SQL filter must not under-include (hide readable rows) *or*
   over-include without the Python re-filter catching it. Test both the admin `*` and `kg:read:*`.
4. Parameterized SQL only; `_like_escape` the wildcard prefix so a permission string can't inject
   LIKE metacharacters. No new checkpointed types (the `atlas_serde()` allowlist is unchanged).
5. The LLM still cannot self-grant — matching is over the principal's policy-derived grants only.
